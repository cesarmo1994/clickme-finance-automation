# OFH-26 - Data Sources And Integrations

## Goal

Define the official data sources and integration boundaries for the ClickMe Admin Portal so the app has clear ownership for every dashboard, workflow, and operational action.

## Source Of Truth Matrix

| Domain | Source of truth | Portal use | Write strategy |
|---|---|---|---|
| Purchase invoices | Azure SQL `finance.dbo.purch_invoices` | Dashboard, invoice review, AP visibility | Written by `InvoicesStorage`; portal can later update review metadata |
| Invoice analytics | Azure SQL views: `invoice_fact`, `vendor_dim`, `date_dim` | Dashboard datasets | Read-only |
| Invoice PDFs and JSON | Finance OneDrive/SharePoint folders | Open document links and trace parser output | Written by automation |
| Vendors | Azure SQL `finance.dbo.vendors` | Vendor normalization, vendor profile, procurement status | Portal-owned after MVP |
| Companies | Azure SQL `finance.dbo.companies` | Company context for invoices and reporting | Admin-owned through portal |
| Payments | Azure SQL `finance.dbo.payments` | AP status and reconciliation | Manual/API-driven later |
| Automation runs | Azure SQL `finance.dbo.automation_runs` | Monitoring and retry history | Automation-owned |
| Users and roles | Microsoft Entra ID plus portal role tables | Authentication and authorization | Entra ID is identity source; portal stores metadata |
| Jira delivery tracking | Jira project `OFH` | Backlog and implementation traceability | Jira-owned |
| Figma designs | Figma file `Finance Invoices Dashboard` | UX reference and dashboard mockups | Figma-owned |
| GitHub source | `cesarmo1994/clickme-finance-automation` | Docs, SQL, app code, infra | GitHub-owned |

## Integration Boundaries

### InvoicesStorage Automation

Responsibilities:

- Watch or process the finance inbox folder.
- Parse PDFs into JSON.
- Insert or update `finance.dbo.purch_invoices`.
- Avoid duplicates through `sha256` and `dedup_key`.
- Move files through the invoice folder workflow.

Not responsible for:

- Portal user permissions.
- Dashboard rendering.
- Manual invoice correction UI.
- Vendor approval workflows.

### ClickMe Admin Portal API

Responsibilities:

- Expose dashboard endpoints backed by `invoice_fact`, `vendor_dim`, and `date_dim`.
- Validate Microsoft Entra ID tokens.
- Enforce portal roles and permissions.
- Provide controlled write endpoints for portal-owned workflows.

Not responsible for:

- Direct PDF parsing.
- Direct mailbox monitoring.
- Replacing the invoice automation.

### Azure SQL

Responsibilities:

- Store operational finance data.
- Provide dashboard-ready views.
- Enforce relational integrity where possible.
- Preserve traceability fields back to source documents and parser payloads.

### Microsoft Entra ID

Responsibilities:

- Authentication.
- Group membership.
- Source identity for portal access.

### GitHub

Responsibilities:

- Version SQL scripts, docs, app code, and deployment assets.
- Keep implementation history.

## Initial API Dataset Contracts

The first dashboard API should read:

- `GET /api/dashboard/invoices` from `dbo.invoice_fact`
- `GET /api/dashboard/vendors` from `dbo.vendor_dim`
- `GET /api/dashboard/dates` from `dbo.date_dim`

The API should not query PDFs, local SQLite, or raw JSON files for dashboard pages.

## Open Questions

- Should invoice PDFs remain in SharePoint/OneDrive paths or move to Azure Blob Storage?
- Should automation runs be written by `InvoicesStorage` immediately or later?
- Should vendor edits write directly to `dbo.vendors` or require approval first?
- Should dashboard exports be generated by the portal API or by SQL/reporting tooling?
